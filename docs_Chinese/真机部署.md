# 真机部署

部署分两端：

- 执行端/RT 控制机：直连 Franka，负责回 home、发布状态、执行 action chunk。
- 推理端/GPU PC：启动策略服务，读取 ZED 和 robot state，把策略输出发回 RT 控制机。

执行端有两条路线：

- x86_64/franky-control：见 `examples/fr3_realtime/rt_machine.md`。
- ARM/C++ libfranka：见 `examples/fr3_realtime_libfranka/rt_machine.md`。

x86_64/franky-control 路线没有做 action 平滑，不推荐用于真机策略执行。优先使用 C++ libfranka 路线，它包含速度 EMA，以及 libfranka 侧速度、加速度、jerk 限幅。

下面命令默认在仓库根目录执行。

## 1. 上机前检查

确认：

- 急停、Franka Desk、碰撞阈值和工作空间都已检查。
- RT 控制机可以 ping 通机器人。
- 推理端可以 ping 通 RT 控制机。
- 两路 ZED 都能被 OpenCV 打开。
- `--prompt` 和训练时任务语义一致。
- 第一次跑新 checkpoint 时，`--request-hz` 先保持 `3.0`，桌面清空，动作幅度从保守配置开始。

## 2. 执行端：回 home

x86_64/franky-control：

```bash
cd examples/fr3_realtime
conda activate fr3-realtime

python go_home.py --robot-ip 172.16.0.2 --time-to-go 8.0
```

ARM/C++ libfranka：

```bash
cd examples/fr3_realtime_libfranka
conda activate fr3-realtime-libfranka

./build/go_home --robot-ip 172.16.0.2 --time-to-go 8.0
```

## 3. 执行端：启动 robot server

x86_64/franky-control：

```bash
cd examples/fr3_realtime
conda activate fr3-realtime

python robot_server.py --robot-ip 172.16.0.2 --bind 0.0.0.0
```

ARM/C++ libfranka：

```bash
cd examples/fr3_realtime_libfranka
conda activate fr3-realtime-libfranka

./build/robot_server --robot-ip 172.16.0.2 --bind 0.0.0.0
```

如果没有夹爪，C++ 版可加：

```bash
--no-gripper
```

推理端后续的 `--robot-ip` 要填 RT 控制机的局域网 IP，不是 Franka 本体的 `172.16.0.2`。

## 4. 推理端：启动策略服务

在 GPU PC 上：

```bash
conda activate openpi-dev

export XLA_PYTHON_CLIENT_PREALLOCATE=false
export XLA_PYTHON_CLIENT_MEM_FRACTION=.70

python ./scripts/serve_policy.py \
  --port 8010 \
  policy:checkpoint \
  --policy.config=pi05_fr3_lora \
  --policy.dir=checkpoints/pi05_fr3_lora/test_train_2/199999
```

如果你已经把 LoRA adapter 合并成完整 checkpoint，也把合并后的目录填到 `--policy.dir`：

```bash
--policy.dir=checkpoints/pi05_fr3_lora_merged/199999
```

LoRA 合并/拆分见 `docs/LoRA合并与拆分.md`。

## 5. 推理端：启动 inference client

另开一个终端：

```bash
conda activate openpi-dev

python ./examples/fr3_realtime/inference_client.py \
  --robot-ip 192.168.1.19 \
  --policy-host 127.0.0.1 \
  --policy-port 8010 \
  --prompt "pick up the cup" \
  --front-camera /dev/video0 \
  --wrist-camera /dev/video2 \
  --flip-wrist \
  --request-hz 3.0 \
  --display
```

参数说明：

- `--robot-ip`：RT 控制机 IP。
- `--policy-host/--policy-port`：策略服务地址。
- `--front-camera`：外部 ZED UVC 节点。
- `--wrist-camera`：腕部 ZED UVC 节点。
- `--flip-front/--flip-wrist`：画面需要 180 度翻转时使用。
- `--request-hz`：向策略服务请求 action chunk 的频率，默认 `3Hz`。

如果要边跑边采集数据，加：

```bash
--record-dir ./datasets/fr3_pick_up_cup_dagger
```

热键：

```text
Enter = 开始一集
s     = 停止并保存当前集
r     = 丢弃当前集
q     = 退出
```

## 6. 常见问题

策略服务连不上：确认 `serve_policy.py` 还在运行，`--policy-port` 一致。如果策略服务在另一台机器，`--policy-host` 填那台机器 IP，并确认端口放行。

一直等 state/camera：确认 RT 控制机 `robot_server` 已启动，推理端 `--robot-ip` 填的是 RT 控制机 IP；确认 `/dev/video*` 编号没变。

画面方向不对：先用 `scripts/zed_preview.py` 看 `LEFT/RIGHT` 和方向，再在 `inference_client.py` 里调整 `--camera-eye`、`--flip-front`、`--flip-wrist`。

动作有明显延迟：降低 `--request-hz` 不会变快，只会减少请求频率。优先检查 GPU 推理耗时、网络、相机读取是否卡顿。日志里的 `infer=...ms` 和 `state_age=...ms` 可以作为判断依据。

动作过激：先停机，检查 checkpoint、prompt、相机视角、joint7 offset 是否和训练一致；C++ libfranka 版还可以降低 `--dynamics-factor` 或调小 `--ema-alpha`。
