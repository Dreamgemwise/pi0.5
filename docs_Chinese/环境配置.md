# 环境配置

这套 FR3 流程通常分成两类机器：

- 推理/训练/采集 PC：有 GPU，跑 OpenPI、策略服务、相机读取、数据采集。
- RT 机器：直连 Franka。数据采集时只读发布 state；真机执行部署时才跑 `robot_server`、`go_home` 和 action 执行。

如果只有一台机器，也可以把推理端和 RT 端放在同一台上，但真机部署时更建议拆开，降低推理显存/相机 IO 对实时控制的影响。

注意区分“数据采集 RT 端”和“真机执行 RT 端”：

- 数据采集 RT 端只能是 x86 主机，使用 FrankaTeleop 创建的 `polymetis` 环境，并运行 `examples/fr3_realtime/readonly_state_publisher.py`。这条链路只读 Polymetis state，不打开 action 端口。
- 真机执行 RT 端才使用 `robot_server`，可以走 x86/franky-control 或 ARM/C++ libfranka。

## 1. 推理/训练 PC：OpenPI 主环境

在仓库根目录创建主环境：

```bash
conda env create -f environment.yml
conda activate openpi-dev
```

`environment.yml` 会安装 OpenPI、`openpi-client`、JAX 训练依赖、OpenCV、ZMQ、msgpack、pyarrow 等。后续训练、启动策略服务、ZED 采集和 LoRA checkpoint 工具都默认在这个环境里跑。

常用检查：

```bash
python -c "import openpi, openpi_client, cv2, zmq, pyarrow; print('openpi env ok')"
```

本项目默认使用本地 `models/pi05_droid` 作为 FR3 LoRA 微调的 base checkpoint。目录应至少包含：

```text
models/pi05_droid/
  params/
  assets/
```

`assets/` 里是 DROID norm stats，FR3 微调和部署都复用它。

## 2. x86_64 RT 控制机：franky-control 版

如果 RT 控制机是 x86_64，并且使用 `examples/fr3_realtime` 里的 Python/franky-control 控制链路：

```bash
cd examples/fr3_realtime
conda env create -f environment.yml
conda activate fr3-realtime
```

确认机器架构：

```bash
uname -m
```

这里应输出 `x86_64`。如果是 ARM/aarch64，不要用这个环境，走下一节的 C++ libfranka 版。

如果这台 x86 机器用于数据采集，不使用 `fr3-realtime` 环境，也不要启动 `robot_server.py`。采集时使用 FrankaTeleop 创建的 Polymetis 环境：

```bash
conda activate polymetis
cd examples/fr3_realtime
python readonly_state_publisher.py \
  --polymetis-host 127.0.0.1 \
  --polymetis-port 50051 \
  --gripper-port 50053 \
  --bind 0.0.0.0
```

`polymetis` 只能运行在 x86 上，所以数据采集 RT 主机不能是 ARM。

如果这台 x86 机器用于真机执行部署，详细上机步骤见：

```text
examples/fr3_realtime/rt_machine.md
```

## 3. ARM RT 控制机：C++ libfranka 版

如果 RT 控制机是 ARM/aarch64，或者你想直接用 C++ `libfranka` 控制 FR3：

```bash
cd examples/fr3_realtime_libfranka
mamba env create -f environment.yml
conda activate fr3-realtime-libfranka
```

然后按下面文档编译 C++ `libfranka` 和 bridge：

```text
examples/fr3_realtime_libfranka/rt_machine.md
```

这条链路仍然使用同一套 ZMQ/msgpack 协议：RT 机 `5555` 发布状态，`5556` 接收 action chunk。它用于真机执行部署，不用于 Polymetis/FrankaTeleop 数据采集链路。

## 4. ZED 相机：UVC/OpenCV 路线

当前代码只取 ZED 的单目 RGB 图像，走 UVC/OpenCV，不需要 ZED SDK、`pyzed` 或系统 CUDA。插上 ZED 后先看 video 节点：

```bash
sudo apt update
sudo apt install -y v4l-utils

v4l2-ctl --list-devices
ls /dev/video*
v4l2-ctl --list-formats-ext -d /dev/video0
```

ZED UVC 原始帧是左右目横向拼接，代码里的约定是 `RIGHT | LEFT`。默认取左目，字段会写成 DROID 风格的：

- `observation.images.exterior_image_1_left`
- `observation.images.wrist_image_left`

预览单个 ZED：

```bash
python scripts/zed_preview.py --camera /dev/video0 --camera-eye LEFT --camera-fourcc YUYV
```

最高分辨率预览：

```bash
python scripts/zed_preview.py --camera /dev/video0 --camera-mode HD2K --camera-eye LEFT
```

`HD2K` 是 `4416x1242@15fps`，单目约 `2208x1242`。实时推理一般先用默认 `HD720`，稳定后再调高。

如果画面左右目不对，试右目：

```bash
python scripts/zed_preview.py --camera /dev/video0 --camera-eye RIGHT --camera-fourcc YUYV
```

如果画面倒置，在采集或推理命令里加：

```bash
--flip-front
--flip-wrist
```

## 5. 网络和端口

推理/采集 PC 需要能连到 RT 控制机：

```bash
ping <RT-PC-IP>
```

RT 控制机需要放行：

- `5555/tcp`：RobotState PUB，推理/采集 PC 订阅。
- `5556/tcp`：ActionChunk PULL，推理 PC 把策略输出发回 RT 机。
- `8010/tcp`：策略服务端口，通常只在推理/训练 PC 本机使用。

如果跨机器启动策略服务，把 `--policy-host` 改成策略服务机器的 IP。

## 6. 常见问题

`franky-control==1.1.3` 找不到：通常是架构不对。x86_64 用 `examples/fr3_realtime`，ARM/aarch64 用 `examples/fr3_realtime_libfranka`。

ZED 打不开：先用 `v4l2-ctl --list-devices` 确认节点，再检查 USB3 连接和 `/dev/video*` 编号。两台 ZED 同时插上时，节点编号可能会变。

策略服务显存不足：启动服务前加：

```bash
export XLA_PYTHON_CLIENT_PREALLOCATE=false
export XLA_PYTHON_CLIENT_MEM_FRACTION=.70
```

训练显存不足：先降低 `src/openpi/training/config.py` 里 `pi05_fr3_lora` 的 `batch_size`，再考虑调小 `XLA_PYTHON_CLIENT_MEM_FRACTION` 或减少并行进程。
